# 测试变更评审

## 规则

当代码可能由代理生成或修复时，测试变更要比实现变更更仔细地阅读。

如果测试被弱化以适配实现，即使测试通过也不够。

真实命令输出比“测试已通过”的声明更有证据价值。

## 标记这些模式

- 删除测试且没有等价替代。
- 弱化断言、放宽匹配器或减少期望值。
- 跳过、禁用、隔离或通过重命名避免执行的测试。
- 过度 mock 掩盖真实行为。
- 高风险行为只有 happy path 测试。
- 测试被改成匹配实现，而不是匹配产品意图。
- bug 修复后删除回归测试。
- 覆盖从公共行为转移到实现细节。
- 覆盖率阈值、lint 规则或必需 gate 被降低。
- fixture、回归样例或反向用例被删除且没有替代。
- 大规模测试重写没有解释旧期望为何错误。

## 高风险验证

对高风险行为，考虑使用 mutation testing 或等价的反向测试，证明实现回归时测试会失败。

当 diff 重写大量测试或 fixture 时，先审查测试意图，再信任实现。

对大规模测试重写，要求解释旧期望为什么错误或过时。如果缺少解释，将实现 review 视为被阻塞或不完整。

## 输出

只要测试有变更，即使未发现问题，也始终输出 `Test Review` 章节。

当测试主导 diff 或断言被重写时，将 `Test Review` 放在实现 findings 前，并说明测试是否仍保护原始产品意图。
