# 评审深度

## L0 / L0

轻量阅读变更文件。明显可用时运行低成本格式或文档检查。除非影响发布或治理，否则不要过度评审生成元数据。

## L1 / L1

评审局部行为和回滚路径。通常一次 AI 评审加人工快速查看即可。

## L2 / L2

评审实现和测试。要求相关验证证据。确认行为变更符合声明意图。

如果验证证据缺失或只是口头声明，先使用 `Not reviewable` 或列出确切缺失证据，再给出合并结论。

## L3 / L3

可行时执行或模拟两次独立评审，并使用不同评审视角：正确性和回归、测试完整性、安全或滥用、运维回滚。

使用多个 AI reviewer 或对比独立评审 pass 时，读取 `references/heterogeneous-reviewers.md`。可用时优先选择不同工具、模型家族、prompt 或评审角色。

可行时要求定向验证。如果无法运行验证，直接说明剩余风险。

对认证、资金、用户数据、删除、生产配置、安全边界或 LLM 工具动作路径，要求有人类负责人接受剩余风险。

## L4 / L4

优先先做方案或设计评审，再做实现评审。可行时拆分大变更。验证不变量、迁移路径、回滚和跨系统兼容性。

对大型混杂 diff、大范围重构、生成代码 churn 或大量测试重写，深度实现评审前要求切片图或更小 PR。

没有充分证据和明确人类负责人时，不要给出可合并结论。
