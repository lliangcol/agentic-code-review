# 风险模型

## 核心维度

按风险分层，而不是按变更由人类还是代理编写来分层。

使用三个维度：

- 影响半径：如果出错，会破坏什么？
- 代码生命周期：这是临时代码、短期代码，还是长期系统代码？
- 共享理解成本：有多少人或系统需要理解或依赖它？

## 风险等级

`L0` 表示文档、注释、格式、生成元数据，或其他不影响运行时行为的变更。

`L1` 表示局部低风险行为，回滚容易，影响半径小。

`L2` 表示普通产品或 API 行为，应该有有意义的测试。

`L3` 表示高影响半径或长期共享代码，包括认证、权限、资金流、用户数据、删除、队列、缓存一致性、基础设施、生产配置和安全边界。

当不可信输入会影响动作、权限、文件、命令、密钥、资金或用户数据时，将 LLM、prompt、检索、agent 循环或工具执行变更按 `L3` 处理。

`L4` 表示架构、跨服务协议、核心数据模型、不可逆迁移、大范围重构、历史改写，或跨多个片段的合并就绪检查。

## 升级规则

按最高风险变更片段决定整体评审等级。一个大 PR 中只要有小范围高风险文件，也应提升评审等级。

当变更制造理解债务时应升级：大量 churn 但没有切片图、重复 helper、混杂关注点、长期共享抽象，或行为隐藏在生成代码后。

当测试被大面积重写、删除、跳过，或被改成匹配实现而不是产品意图时，应升级。
