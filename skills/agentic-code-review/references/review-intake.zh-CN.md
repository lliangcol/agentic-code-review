# 评审入口

## 目的

在投入深度评审前，先判断变更是否具备评审条件。

`Not reviewable` 表示提交物缺少负责任评审所需的证据或形态。它不同于 `Needs confirmation`，后者表示评审后仍有产品、负责人或上下文事实不确定。

## 必要证据

深度评审前，先寻找以下输入：

- 意图：变更目的是什么，哪些行为应该改变。
- 范围：受影响的文件、模块、服务或用户流程。
- 大范围变更的切片图：行为片段、生成文件、机械 churn，以及需要真实评审的文件。
- agent 生成变更的决策日志：执行的方案、关键假设、被排除的替代方案和需要人类判断的点。
- 风险说明：预期 `L0-L4` 等级或作者的风险理由。
- 验证证据：运行过的命令和相关输出，而不仅是“测试通过”。
- 测试变更说明：测试为何变更，以及它们保护什么行为。
- `L2+` 变更的回滚或缓解说明。
- `L3+` 或合并就绪检查的人类合并负责人。

## 不可评审信号

当负责任的评审主要是在补建缺失的意图或证据时，使用 `Not reviewable`。

- diff 过大或混杂，无法作为一次评审检查，且没有提供方案或切片图。
- 作者要求判断可合并，但没有验证证据。
- 非平凡的 agent 生成变更缺少决策日志，导致评审者必须从 diff 中重建意图。
- 测试被大面积重写，但没有解释旧期望为何错误。
- 生成代码、第三方代码或机械 churn 掩盖了行为变更。
- 高风险区域发生变更，但没有人类负责人或回滚路径。
- LLM、prompt、检索或工具执行行为变更，但没有说明信任边界。

## 评审者动作

对 `Not reviewable`，列出开始评审所需的最小缺失证据。

不要把缺失证据转写成猜测性问题。

当低成本信号表明变更可能过大、混杂、高 churn 或证据不足，无法负责任地深度评审时，读取 `references/review-effort-signals.md`。

如果只有部分变更可评审，就评审该部分，并将其余部分标记为 `Not reviewable`。
